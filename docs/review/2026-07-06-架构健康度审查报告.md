# 架构健康度审查报告

- 日期：2026-07-06
- 审查方式：5 路并行探查（层边界依赖、单一事实源、超大组件、退役残留、语义模型契约成色），全部结论带 file:line 证据
- 结论有效性：以当日 main 分支代码为准

## 0. 总体判决

**不需要推倒重构。** 骨架方向正确，小尺度纪律优秀（172 个 JS 文件平均 169 行、无千行巨兽、无深嵌套、无垃圾抽屉模块、无循环依赖、JSX 执行端严格守边界）。

**但三个核心架构承诺目前只靠自觉，没有强制力，且已查到真实病变：**

1. "统一语义模型是共享事实层" —— 半真。模型形状无权威定义，HTML 侧与 InDesign 侧实际产出两种方言。
2. "协议注册表是字段唯一事实源" —— 半真。注册表只做事后校验，不是各处代码取用的源头；已有 3 处白名单真实漂移。
3. "adapters → semantic-model → writers 单向分层" —— 目录命名层面存在，require 图上未被执行；存在双向直连和依赖倒置。

另有一类**按仓库硬规则应立即按严重缺陷处理**的急症：无丢失硬门槛存在"双侧空值互相抵消"的假通过路径，以及一处被测试固化的 legacy 死分支。

## 1. 健康面（确认无病，不需要动）

- 尺寸纪律：>400 行文件仅 13 个（约 6%），多数是"归一化整份快照/枚举协议字段/审计双向 diff"类单一职责长实现，属健康长文件。
- 缩进/函数纪律：全仓库未发现超 6 层嵌套；早返回扁平风格执行彻底。
- `src/shared/` 按关注点切 7 个小文件，是良好共享层，不是 utils 垃圾抽屉。
- `src/semantic-model/` 与 `src/protocol/` 均无反向依赖业务层。
- `_indesign_scripts/` 执行端确认只做 instructions 分发执行与原生事实抽取，无 HTML/CSS 解析、无语义推理（唯一例外见 §5 的 extract_blueprint.jsx）。
- 退役目录 `src/paged-html/`、`src/indesign-reverse/` 确已删除；`retired.js` 字段、blueprint 迁移、`legacy-label` 观察标签均为文档承认、显式报错的合规兼容边界。
- 文档-代码同步度高：AGENTS.md 第 9 节命令全部真实存在；插件 tools 目录与声明一致；无幽灵路径。

## 2. 急症（P0：按硬规则属严重缺陷，小工作量，立即修）

### 2.1 无丢失硬门槛存在假通过路径

- `src/writers/html/audit/content-inventory.js:47-53、182-190`：`expected.pages || []` 与 `actual.pages || []` 双侧兜底互相抵消——若上游 `deck.config.json` 缺失 `pages`，比对循环一次不跑、`errors` 恒空、`ok` 恒 true，把"根本没比较"伪装成"内容无丢失"。content-inventory 是 AGENTS.md 9.4 明文的无丢失硬门槛。
- `content-inventory.js:231-258`：`readAssetAliases` 解析失败静默返回空表，无 warning，资源比对可能误判。
- `scripts/audit-effective-diff.js:129-132`：CLI 出口 `catch(_){}` 吞掉 JSON 解析失败后 exitCode 保持 0——门禁自身出错时报"成功"。
- 同类模式另见 `structure-signature.js:32-33、51-52`、`reverse-snapshot-structure.js:828-841`（矢量双侧缺失判"未变"）、`parent-page-furniture.js:230-231`。

修法：空输入必须显式 fail（如 `CONTENT_INVENTORY_INPUT_INVALID`），门禁出口无法判定时 exitCode 非 0。

### 2.2 被测试固化的 legacy 死分支

- `_indesign_scripts/lib/hi_executor.jsxinc:6-52`：`HI.runLegacyBuildInstructions` / `HI.runPagedHtmlBuildInstructions` 使用明令禁止的 `legacy` 与退役架构名 `pagedHtml` 命名；错误提示引用已不存在的 `pagedHtml.compileInstructions`；`test/indesign-executor/executor-script-static.test.js:63` 还把该 legacy API 断言为必须存在。全仓库无任何来源会生成触发 legacy 分支的旧 schema。
- 修法：删除 legacy 分支与断言，成功路径函数改用当前架构命名。

### 2.3 孤儿模块

- `src/adapters/html/reader/stacking.js`：无人引用；`browser-snapshot.js:167` 本地重写了同功能函数。
- `src/adapters/html/reader/style-reader.js`：无人引用，且其样式属性清单与实际使用的 `browser-style-capture.js` 已分叉——事实上的第二能力矩阵。
- 修法：直接删除。

## 3. 结构病一：语义模型是两种方言（P1，最高杠杆）

证据（对比两侧 normalizer 出口）：

| 概念 | HTML 侧 | InDesign 侧 |
| ---- | ---- | ---- |
| 原始类型 | `items[].type`（未登记 registry） | `items[].sourceType`（已登记） |
| 无语义默认值 | `null` | 字符串 `'unknown'` |
| styleRefs 键集合 | 透传 + synthesized* | 硬编码 paragraphStyle/objectStyle/frameStyle/tableStyle |
| bounds | 经 `semantic-model/layout.js` 统一计算 | 直接透传，不走共享计算 |

后果链：

- `validateSemanticModel` 是 semantic-model 层唯一公共 API，但**生产链路零调用**（仅测试使用）；两个 adapter 出口都不校验。
- writers 用约 125 处 `&&` 链 + 约 668 处 `||` 兜底吸收方言差异；出现 `item.role || item.type || item.sourceType || 'unknown'` 式三选一读取（`semantic-reconstruction/object-graph/node-facts.js:21`、`author-audit.js:162`）。
- `semantic-reconstruction` 虽然依赖方向干净，但内部逻辑只认 InDesign 方言字段（`labelStatus`/`rejectionReasons`），并非真正"处理统一模型任意实例"。
- format extension 只是 registry 里的分类标签，模型对象上 `effects`/`textFrameStyle` 等直接与 canonical 字段平铺同级，无 `extensions.*` 结构隔离。

修法方向（一场"契约加固"，非重写）：

1. 裁定每个分歧字段的 canonical 形态（登记进 registry），两侧 adapter 对齐产出。
2. 在两个 adapter 出口强制调用 `validateSemanticModel`（默认 strict，失败即抛），把校验从测试脚手架升级为运行时门禁。
3. 契约可信后，逐步剥除 writers 的三选一读取与兜底（每剥一处配测试）。
4. format 专有字段迁入 `extensions.indesign.*` 结构命名空间。

## 4. 结构病二：注册表不是源头，白名单已漂移（P1）

- 47 个文件硬编码 `data-id-*` 字符串字面量；`src/protocol/` 不导出属性名常量，全仓库零文件从 registry 取属性名；`capabilityFor`/`assertWritable` 在 protocol 外零调用——能力矩阵对运行时写出行为不承载。
- **已漂移的三处（确凿矛盾）：**
  - svg 角色分类：`adapters/html/reader/candidate-elements.js:3-18` 判为 `graphic`，`writers/html/audit/content-inventory.js:307-313` 判为 `shape`。
  - `data-id-role` 合法值：registry 无枚举；`candidate-elements.js:23-25` 列 7 值，`authoring-validator.js:422-424` 只列 4 值（缺 line/background/decoration），而 `background/decoration` 正被 writers 实际写出。
  - SAFE_TAGS：`author-render-utils.js:1-8` 与 `visual-html-writer.js:23-53` 两份白名单集合已不一致。
- 重复工具（暂未矛盾但必然漂移）：`normalizeText` 8 处定义 3 种语义；`safeClass` 族 5 处实现行为各异（直接影响 synth token → CSS class 链路）；单位换算 3 份拷贝其中 `authoring-validator.js:273-289` 换算基准与另两份语义不同；NAS 路径逻辑在 `content-inventory.js` 重复实现未复用 `shared/nas-paths.js`。
- 语义样式 kind 枚举在 `semantic-preset/maps.js`、`semantic-preset/audit-authoring.js`、`label-whitelist.js` 三处独立维护，且存在单复数命名分裂。

修法方向：

1. `src/protocol/` 导出属性名常量表、role 枚举、tag 白名单；先收编三处已漂移清单（漂移本身要先裁定谁对）。
2. 机械扫荡 47 个文件改为引用常量。
3. 加一条静态测试：src（protocol 外）/scripts 中禁止出现裸 `data-id-` 字符串字面量——防 GPT 复发。
4. `normalizeText`/`safeClass`/单位换算收敛到 `src/shared/`（text.js 等），同名不同义的先改名再合并。

## 5. 结构病三：分层是目录约定，不是 require 事实（P1）

- **双向直连**（两两格式直连，AGENTS.md §5 明令禁止）：
  - `adapters/html/normalizer/snapshot-to-model.js:1` → require `writers/indesign/style-compiler`（归一化依赖目标写出器）。
  - `writers/indesign/instructions-compiler.js:1` → require `adapters/html/normalizer/snapshot-to-model`（且被 index.js 作为公共 API 再导出、被测试固化）。
  - `writers/indesign/graphic-instructions.js:2` → require `adapters/html/reader/asset-detector`。
- **依赖倒置**：`src/indesign-cli-plugin/tools/reverse-export.js:3` → require `scripts/indesign-reverse-export.js`——正式插件工具依赖内部脚本文件；`tools/authoring-lint.js` 绕开 `src/authoring` 公共入口直连内部文件。
- **主链路只在脚本里组装**：`semantic-reconstruction` 在 src 内零生产调用，AGENTS.md 声明的反向链路唯一组装点是 `scripts/indesign-reverse-export.js`；scripts 中 4 个文件（占 scripts 总行数约 63%）含 src 无对应模块的业务逻辑——`audit-conversion-gate.js`（364 行门禁算法全在脚本）、`audit-reverse-visual.js`（用 Playwright 平行重写了一套浏览器几何捕获，与 `browser-snapshot.js` 职责重合）、`indesign-e2e.js`（805 行含多个审计函数）、`indesign-reverse-export.js`。
- **隐藏第三层**：`src/authoring`、`src/semantic-preset`、`src/shared` 被各层大量依赖，却不在 AGENTS.md §4/§5 的地图与边界规则里。

修法方向：

1. 反向流水线组装函数上移到 src（如 `src/writers/html/reverse-pipeline.js` 或独立编排模块），scripts 与插件都改调它。
2. conversion-gate 判定算法迁入 src 审计层，脚本只接线；`audit-reverse-visual` 的几何捕获合并回 `adapters/html/reader`。
3. 拆双向直连：`instructions-compiler` 的"快照→模型→指令"组合入口若确有必要保留，应上移到明确的编排层，而不是让 writer require adapter；`style-compiler`/`asset-detector` 中被两侧共用的纯函数下沉 `src/shared/` 或经模型字段传递。
4. AGENTS.md 补登 `src/authoring`、`src/semantic-preset`、`src/shared` 的边界规则。
5. 加依赖方向静态测试：解析 src 下 require 图，断言 adapters 不得 require writers、writers 不得 require adapters、src 不得 require scripts——防回退护栏。

## 7. 修复优先级与顺序建议

| 级别 | 内容 | 工作量 | 理由 |
| ---- | ---- | ---- | ---- |
| P0 | §2 三项急症（门禁假通过、legacy 死分支、孤儿模块） | 小（1-2 天量级） | 仓库硬规则定性为严重缺陷；门禁可信度是一切审计（含可编辑性基准判定器）的地基 |
| P1-a | §3 模型方言统一 + 出口强制校验 | 中 | 最高杠杆：直接减少 668 处兜底的存在理由，也是基准判定器消费报告的可信前提 |
| P1-b | §4 注册表升级为源头 + 三处漂移收编 + 防复发静态测试 | 中 | 漂移已造成事实矛盾（svg 分类），越拖越贵 |
| P1-c | §5 反向链路上移 src + conversion-gate 入 src + 依赖方向护栏测试 | 中 | 解除插件对 scripts 的依赖倒置，让声明架构与 require 图一致 |
| P2 | §6 卫生项 | 小，可随手 | 触及对应文件时顺带处理 |

护栏原则：每项 P1 完成时必须落一条**静态测试**（依赖方向断言、裸字面量扫描、出口校验必跑），把架构规则从文档承诺变成 `npm test` 的一部分——否则同类问题会被后续执行者再次引入。

## 8. 与可编辑性基准计划的关系

`docs/superpowers/plans/2026-07-06-agent-editability-benchmark-implementation-plan.md` 的判定器直接消费 `content-inventory` 与 `structure-signature` 报告。**P0 中的门禁假通过修复应排在基准实施之前或并行**，否则基准批改机会继承同一批"双侧空值抵消"盲区，判定结果不可信。
