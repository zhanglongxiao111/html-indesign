# 测试增长质量审计报告

## 当前结论

当前结论：**存在 guardrail 假绿风险**。

判断不是“整批低价值测试”。本轮新增测试里有一批真实验证运行行为、CLI 失败路径和生成物同步的高价值测试；但 `test/architecture/` 里至少有两处 guardrail 仍可被“写空测试名”或“加 baseline 豁免”绕过，另外 `G4` 还有并发扫描脆弱性。综合风险等级：**P1**。

## Findings

### 1. [P1] `G4.1` 只校验“测试名存在”，允许空测试体假绿

证据：

- `test/architecture/audit-fail-closed.test.js:76-88` 明确把一个空测试体当成合格覆盖：
  - `test("audit-covered invalid-input 必须 fail", () => {});`
  - 随后 `assert.deepEqual(violations, []);`
- `test/architecture/audit-fail-closed.test.js:110-122` 对 destructured import 也是同样的空测试体。
- `test/architecture/audit-fail-closed.test.js:143-149` 的实际 guardrail 只检查 `expectedName` 是否出现在 `testNames` 集合里。

影响：

- 这不是在验证“invalid-input 测试真的断言了失败行为”，而是在验证“仓库里存在一个叫这个名字的测试”。
- 虽然 `G4.2` 还会在 `test/architecture/audit-fail-closed.test.js:152-157` 直接空参执行脚本并拒绝 `status === 0`，但 `G4.1` 自身仍然是弱扫描，可被表面测试满足。
- 这直接违反“guardrail 不能靠弱扫描假绿”的项目原则。

结论：

- 当前仓库里的多数具体 audit CLI 测试是实测的，例如 `test/indesign-to-html/conversion-gate-cli.test.js:194-253` 会真实起进程并校验 `CONVERSION_GATE_INVALID_INPUT`，所以**现状不是全盘失效**。
- 但 `G4.1` 这条 guardrail 本身的实现质量不够，属于**可绕过的 guardrail**。

### 2. [P1] 除 `G3` 外，其余 architecture baselines 都能通过“新增 exemptions”把红灯改绿

证据：

- `test/architecture/helpers/baseline-ratchet.js:13-22` 允许任何 baseline exemption 生效，只要求有 `reason` 和 `cleanupRef`。
- `test/architecture/dependency-direction.test.js:256-264`
- `test/architecture/protocol-literals.test.js:76-84`
- `test/architecture/audit-fail-closed.test.js:125-133`
- `test/architecture/retired-naming.test.js:67-74`
- `test/architecture/single-implementation.test.js:138-145`
- `test/architecture/docs-code-sync.test.js:66-72`
- `test/architecture/orphan-modules.test.js:115-122`

以上 guardrail 都只是“当前结果与 baseline 相等即通过”。

对照：

- 只有 `test/architecture/semantic-model-contract.test.js:380-390` 额外校验了“baseline 不得比起始点新增 exemptions”。
- baseline 文件目前全部为空：
  - `test/architecture/baselines/G1.json:1-3`
  - `test/architecture/baselines/G2.json:1-3`
  - `test/architecture/baselines/G3.json:1-3`
  - `test/architecture/baselines/G4.json:1-3`
  - `test/architecture/baselines/G5.json:1-3`
  - `test/architecture/baselines/G6.json:1-3`
  - `test/architecture/baselines/G7.json:1-3`
  - `test/architecture/baselines/G8.json:1-3`

影响：

- 现在空 baseline 本身不是问题，当前 guardrail 也确实在报真实结果。
- 问题在于：**未来任何人都可以把新违规直接写进 G1/G2/G4/G5/G6/G7/G8 baseline，再补两句 reason/cleanupRef，测试仍然全绿。**
- 这属于典型的 baseline 型假绿通道。

结论：

- `G3` 已经证明更强做法是可实现的。
- 其余 architecture guardrail 还没有同等强度的 baseline 反逃逸约束。

### 3. [P2] `G4` 扫描整个 `test/`，对 `test/workspace/` 临时目录并发变化不稳

证据：

- `test/architecture/audit-fail-closed.test.js:178-189` 会递归收集整个 `test/` 下的 `.test.js`。
- `test/architecture/audit-fail-closed.test.js:445-452` 的 `collectFiles()` 直接 `readdirSync()` 递归，没有 `ENOENT` 容错，也没有排除 `test/workspace/`。
- 本次只读审计里，将

```powershell
node --test test/architecture/*.test.js
node --test test/protocol/*.test.js
```

并发外部运行时，`G4 current audit fail-closed violations match the ratchet baseline` 失败，报：

```text
ENOENT: no such file or directory, scandir '...\\test\\workspace\\generated-docs-stage-11'
```

- 随后单独顺序重跑 `node --test test/architecture/*.test.js`，`61/61` 通过。

影响：

- 这说明 `G4` 的仓库扫描器对测试过程中的临时目录生命周期敏感。
- 它不是 guardrail 假绿，而是**guardrail 自身存在并发脆弱性**，会制造偶发红灯，最终诱导开发者降低信任或绕过测试。

## 可删候选

以下仅属于**有条件可删**，前提是先把 `G4.1` 从“测试名扫描”替换成“真实失败行为/断言验证”：

1. `test/architecture/audit-fail-closed.test.js:76-88`
   - `G4 accepts a real Node test case named for invalid-input coverage`
   - 价值仅在证明“名字命中即可过关”。
2. `test/architecture/audit-fail-closed.test.js:110-122`
   - `G4 accepts a destructured node:test import for invalid-input coverage`
   - 同样只是在喂养当前弱扫描器。
3. `test/architecture/audit-fail-closed.test.js:54-74`
   - `G4 ignores invalid-input coverage text in comments and ordinary strings`
   - 如果 `G4.1` 不再依赖名称解析，这类 parser-shape 测试可整体退役。

不建议直接删除的原因：在当前实现下，这几条测试仍然是 `collectNodeTestNames()` 行为的唯一回归保护。

## 不可删理由

以下新增测试应保留：

1. `test/indesign-to-html/conversion-gate-cli.test.js`
   - 同时覆盖 src 模块直调、CLI 进程、预算通过、预算失败、trusted-source 回归、parseable 但缺字段输入。
   - 这是实打实的 fail-closed 行为测试，不是文本扫描。
2. `test/protocol/current-field-facts.test.js`
   - 直接调用真实 `placementFromAttributes()` 与 `fieldRegistry`，验证当前代码行为与退役字段边界。
   - 这类测试直接对“当前事实源”负责。
3. `test/protocol/generated-docs.test.js`
   - 同时校验生成器确定性、生成文档字节级同步、CLI 缺参失败。
   - 能防止“文档更新漏跑生成器”这种假成功。
4. `test/architecture/semantic-model-contract.test.js`
   - 虽然文件过大，但它真实调用 adapter/runtime 路径做 `null` 输入失败和 registry surface 对照，价值高于体积问题。

## 是否需要沉淀为长期规范

**需要。**

建议沉淀为长期规范的三条规则：

1. guardrail 不得只校验测试名、注释文本或字符串命中；必须校验真实失败行为、退出码、错误码或输出结构。
2. baseline 只能记录经批准的历史债务；若 baseline 初始为空，后续新增 exemption 必须有额外防线，至少达到 `G3` 这种“禁止静默扩 baseline”的强度。
3. 仓库级扫描器不得递归依赖 `test/workspace/` 这类临时目录；应优先基于 `git ls-files`，或对 `ENOENT` 与临时输出目录显式容错/排除。

## 总结

这轮测试增长里，`protocol` 与 `conversion-gate` 方向的大部分新增测试是高价值的，确实在测真实行为；真正的薄弱点主要集中在 architecture guardrail 自身的防逃逸设计，而不是“测试写得太多”。

如果只问一句最终判断：**不是空测试堆砌，但 architecture guardrail 仍然存在 P1 级假绿风险，需要继续硬化。**
