## 当前结论

结论：**可疑增长，且发现退役治理未收口**。

判断依据不是“代码多了”，而是两处协议生命周期机制没有收口：

1. `retired.modelPaths` 已进入 registry entry schema，但 registry/query/validator 公开面仍只支持 retired HTML attr。
2. `items[].effects` / `items[].textFrameStyle` 已在描述和 migration 元数据里被称为 retired，但没有正式进入 retired registry。

同时，主干新增里也有一批应保留的必要硬化：

- `src/protocol/constants.js` 把 `HTML_DATA_ID_ATTRIBUTES`、`ITEM_ROLE`、`STYLE_KIND`、`SYNTHESIZED_STYLE_KIND` 从 registry 派生，减少散落字面量。
- `src/semantic-model/validator.js:35-48` 改为把完整 `DocumentModel` 传给 `validateModelFields`，让 allowedValues 校验能拿到真实值。
- `src/protocol/scanners/model-surface-paths.js` + `src/protocol/scanners/scan-model-paths.js` 虽然体量大，但确实在把未知字段暴露为 fail-closed 校验，而不是静默漏掉。

风险总评：**P1**

## Findings

### [P1] `retired.modelPaths` 只接入了 entry schema，没有接入 registry/query/validator 公开面

**结论**

这不是“后续再补”的小缺口，而是已经把“retired model path”作为协议能力引入，却没有把查询和校验链条同步收口。现在 `items[].type` 这种 retired model path 可以被 `validateModelFields` 识别为 retired，但不能被统一 lifecycle 查询接口读取，也没有 retired model path 专用 validator surface。

**证据**

- `src/protocol/field-entry.js:12,150-256`
  - 新增 `RETIRED_MODEL_READ_POLICIES`
  - 新增 `retired.modelPaths` 校验分支
- `src/protocol/fields/retired.js:2-24`
  - 把 `items[].type` 注册为 `lifecycle: 'retired'`
  - `replacedBy: 'items[].sourceType'`
- `src/protocol/registry.js:17-66`
  - 只建立了 `byRetiredHtmlAttr`
  - 没有 retired model path index / lookup
- `src/protocol/field-query.js:53-121`
  - `lifecyclePolicyFor()` retired 分支只读取 `retired.htmlAttrs`
  - 对 `retired.modelPaths` 会抛 `LIFECYCLE_POLICY_INVALID`
- `src/protocol/validators/validate-retired-fields.js:1-47`
  - 输入面只有 `htmlAttrs`
  - 没有 retired model path 校验入口
- 实测：

```text
node -e "const { fieldRegistry, lifecyclePolicyFor } = require('./src/protocol'); ..."
=> retired.htmlAttrs.dataIdPage => 正常返回 policy
=> items[].type => ERROR:LIFECYCLE_POLICY_INVALID:items[].type
```

- 测试覆盖也停留在 retired HTML attr：
  - `test/protocol/lifecycle.test.js:41-171`
  - `test/protocol/validate-retired-fields.test.js:8-132`
  - 这两组都没有覆盖 retired model path 查询

**影响**

- registry 不再是统一退役事实源：同样是 retired field，HTML attr 可统一查询，model path 不可统一查询。
- 退役模型路径只能靠 `validateModelFields` 的特例警告存活，无法通过 `lifecyclePolicyFor` 或 retired validator 走同一条治理链。
- 这会诱发后续继续写旁路逻辑，直接破坏“协议字段必须集中登记”和“退役字段不得留在结构化链路里靠隐式兼容活着”的项目原则。

### [P1] `items[].effects` / `items[].textFrameStyle` 被写成“retired”，但没有正式登记为 retired 字段

**结论**

这两条旧平面路径当前只是“文案上 retired、运行时按 unknown 拒绝、文档里只留 migration 备注”，并没有进入 retired registry。结果是：`items[].type` 有正式生命周期，`items[].effects` / `items[].textFrameStyle` 没有，退役治理标准不一致。

**证据**

- `src/protocol/fields/reverse-surfaces.js:87-111`
  - `items[].extensions.indesign.effects` 描述里写明 “Flat items[].effects is retired”
  - `items[].extensions.indesign.textFrameStyle` 描述里写明 “Flat items[].textFrameStyle is retired”
- `src/protocol/fields/retired.js:1-114`
  - 没有对应 `items[].effects` / `items[].textFrameStyle` retired entry
- 实测 registry lookup：

```text
items[].effects => MISSING
pages[].items[].effects => MISSING
items[].textFrameStyle => MISSING
pages[].items[].textFrameStyle => MISSING
```

- 生成文档现状：
  - `docs/规范/PROTOCOL_FIELD_REGISTRY.md:332-333`
    - 只把两者写成 active extension 的 `migration=...`
  - `docs/规范/PROTOCOL_FIELD_REGISTRY.md:513`
    - retired model path 只有 `items[].type`
    - 没有 `retiredModelPath=items[].effects`
    - 没有 `retiredModelPath=items[].textFrameStyle`
- 测试也把它们定义成 unknown，而不是 retired：
  - `test/protocol/validate-model-fields.test.js:1088-1117`
  - 测的是 strict validation 报 `MODEL_FIELD_NOT_REGISTERED`，不是 `MODEL_FIELD_RETIRED`
- `test/protocol/current-field-facts.test.js:222-240`
  - 只验证 `migration.from/to`
  - 没验证 retired lifecycle

**影响**

- 使用方无法区分“这是真正登记过的退役字段”还是“只是扫描到的陌生字段”。
- 生成的协议字段文档内部语义不一致：文案说 retired，但 retired section 没有它。
- 这会让后续 adapter/writer 在治理旧路径时继续走 description/migration 文本旁路，而不是走正式 lifecycle。

## 可删候选

这里**没有可直接低风险删除**的运行时代码。当前更像“半接入未收口”，不是“新增全都多余”。

只有以下**条件性可删候选**：

1. `src/protocol/fields/reverse-surfaces.js:87-111` 的 “retired” 文案式 migration 备注
   - 为什么可删：如果短期内不准备把 `items[].effects` / `items[].textFrameStyle` 正式接入 retired registry，这种“文案宣称 retired、机制上却不是 retired”的写法会持续制造第二套语义。
   - 删除风险：会失去迁移线索，降低历史可读性。
   - 建议验证：
     - `node --test test/protocol/current-field-facts.test.js test/protocol/generated-docs.test.js`

2. `src/protocol/field-entry.js:150-256` 新增的 `retired.modelPaths` schema 分支
   - 为什么可删：如果团队决定本阶段只支持 retired HTML attr，不支持 retired model path，那就不该把半套 schema 先暴露出来。
   - 删除风险：会回退 `items[].type` 的正式 retired 语义，`validateModelFields` 会失去当前的 retired 分类能力。
   - 建议验证：
     - `node --test test/protocol/current-field-facts.test.js test/protocol/validate-model-fields.test.js`

我的判断是：**不建议删**。更合理的是把后续 query/validator/registry surface 补齐。

## 不可删理由

1. `src/protocol/constants.js:9-49`
   - 这是本轮最干净、最应保留的增长之一。
   - 它把 `HTML_DATA_ID_ATTRIBUTES`、`ITEM_ROLE`、`STYLE_KIND`、`SYNTHESIZED_STYLE_KIND` 从 registry 派生出来，避免继续散落第二份字面量清单。
   - `test/protocol/constants.test.js:55-180` 已覆盖其 fail-closed 行为。

2. `src/protocol/item-role-helpers.js:1-61`
   - 角色子集来自 registry-derived constants，而不是散落常量。
   - 这是把 HTML physical role / authoring-mappable role 收口到统一协议值域的必要步骤。

3. `src/protocol/scanners/model-surface-paths.js:5-419` + `src/protocol/scanners/scan-model-paths.js:12-600+`
   - 体量大，但它们是 strict field validation 能 fail-closed 的前提。
   - 当前 unknown/retired/styleRefs/table/text/visualStyle 的拒绝行为都依赖这些 scanner。
   - `test/protocol/validate-model-fields.test.js` 和 `test/semantic-model/validator.test.js` 已覆盖大部分运行面。

4. `test/architecture/semantic-model-contract.test.js`
   - 这是明显的大体量测试文件，但它属于 guardrail，而不是运行态复杂度外溢。
   - 当前不建议删；更合理的是后续按规则拆分，而不是回退 guardrail。

## 是否需要沉淀长期规范

**需要。**

建议沉淀到 `docs/规范/PROTOCOL_FIELD_REGISTRY.md` 的生成规则说明，或更上层的协议规范中，至少明确两条：

1. **retired model path 不是描述性元数据，必须具备与 retired HTML attr 对等的 registry lookup、query API、validator API 和测试覆盖。**
2. **`migration.from` 不能替代 retired lifecycle。** 只要文档或描述里使用 “retired” 一词，就必须在 `src/protocol/fields/retired.js` 有正式 entry，并进入生成文档 retired section。

## 总结

- 主体增长里，`constants.js`、role helper、strict field validation、scanner 扩面，整体是必要硬化。
- 但退役治理只收了半套：**`retired.modelPaths` 已引入 schema，未引入统一查询和验证；旧平面 reverse 路径被称为 retired，却未正式登记 retired。**
- 因此本轮结论不是“必要增长已完全收口”，而是：**必要增长为主，但存在 2 个 P1 级治理缺口，不能算协议生命周期已经闭合。**
