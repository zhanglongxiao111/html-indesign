# 汇总：架构硬化核心代码膨胀审计

## 背景

用户关注本轮架构硬化重构后核心代码净增是否可能来自老代码未退役。本文由主控汇总各子 Agent 只读审计报告后更新。

## 当前结论

已完成 7 个 `gpt-5.4 xhigh` 只读子 Agent 并发审计。主控交叉核验后，结论如下：

1. **核心代码大幅增长的主因不是旧 `paged-html` / `indesign-reverse` / public facade 没退役。**
   - `src/paged-html`、`src/indesign-reverse` 已不存在。
   - public API 不再暴露 `pagedHtml`、`indesignReverse`、`historicalTemplate`。
   - G5 retired naming、G8 orphan modules、public API 退役测试均通过。

2. **确实存在需要继续清理的“半收口/重复实现/门禁假绿”问题。**
   - 这些问题不是旧大模块原封不动残留，而是新硬化层落地后仍有边界不够完整。
   - 严重级别最高为 P1，应进入下一轮修复计划。

3. **最明确的非产品性膨胀是 `.superpowers/sdd/*.md` 已被跟踪。**
   - `.gitignore` 已忽略 `.superpowers/`。
   - `git ls-files .superpowers/sdd/*.md` 仍列出 14 个已跟踪过程报告。
   - 这部分净增约 1363 行，应从版本库删除；需要长期保留的报告应放 `docs/AI协作/`。

## P1 问题清单

### P1-1：协议 retired model path 治理半接入

证据：

- `items[].type` 在 registry 中是 retired，但 `lifecyclePolicyFor(fieldRegistry, 'items[].type')` 抛 `LIFECYCLE_POLICY_INVALID`。
- `items[].effects` / `items[].textFrameStyle` 在 active extension 描述里称 retired，但 `fieldRegistry.getByPath(...)` 返回空。

影响：

- 退役字段治理不一致：HTML attr 可统一查询，model path 不可统一查询。
- `migration.from` 和描述文本有替代 lifecycle 的风险。

建议：

- 为 retired model path 补齐 registry index、query API、validator/test。
- 将 `items[].effects` / `items[].textFrameStyle` 正式登记为 retired model path，或移除“retired”描述避免第二语义。

### P1-2：reverse export 与 E2E 的 author audit 强弱双路径

证据：

- `src/reverse-pipeline/index.js` 只调用低阶 `src/writers/html/audit/author-audit.js`。
- `scripts/indesign-e2e.js` 另行调用 `src/writers/html/audit/reverse-roundtrip.js` 的 source roundtrip、content inventory、structure signature、second-pass stability。

影响：

- 同样的 reverse author package，在 reverse export 路径可能提前成功，E2E 才失败。
- 违反“失败要早于假成功”和“禁止强弱双路径”。

建议：

- 把高阶 author audit 接到 `src/reverse-pipeline`，让 CLI、plugin reverse export、E2E 复用同一 `src` 级门禁。

### P1-3：G4/G6 等 guardrail 存在假绿通道

证据：

- G4.1 只检查测试名是否包含 `invalid-input 必须 fail`，空测试体也能通过。
- 除 G3 外，多数 baseline guardrail 只要求 actual 等于 baseline；未来新增 exemption 即可变绿。
- G6 只扫固定 helper 名字，挡不住改名后的同义重复。

影响：

- 架构护栏当前能抓住已有问题，但抗绕过能力不足。
- 后续维护者可能通过 baseline 或改名绕过规则。

建议：

- G4 改为验证真实失败行为或断言结构，不只扫测试名。
- 所有 G1/G2/G4/G5/G6/G7/G8 baseline 增加“禁止新增 exemption”或显式批准机制，至少达到 G3 强度。
- G6 从固定函数名扩展到语义族：资产路径、文本归一化、table width、border 可见性等。

### P1-4：shared/style-synthesis 抽取后仍有 helper 重复

证据：

- `src/writers/html/asset-reference-policy.js`、`author-asset-packager.js`、`author-resource-paths.js` 各自有路径 key / remote / sanitize helper，且 `file:` 语义已有漂移。
- `src/style-synthesis/index.js` 与 `src/writers/indesign/table-instructions.js` 同名 `normalizeTableWidths` 行为不同。
- `src/style-synthesis/box-model.js` 与 `src/writers/indesign/instruction-writer.js` 各有 `visibleBorder`。
- `src/shared/text.js` 已有文本归一化，但 `instruction-writer.js` 仍有本地 `normalizeInstructionText`。

影响：

- 不是旧模块未退役，而是新边界还没有做到真正单一来源。

建议：

- 优先裁定并统一资产路径 helper。
- 再收敛 table width、visible border、instruction text normalization。
- 同步升级 G6，避免改名重复继续漏过。

### P1-5：`.superpowers/sdd/*.md` 过程报告被版本库跟踪

证据：

- `git ls-files .superpowers/sdd/*.md` 列出 14 个已跟踪报告。
- `.gitignore` 已忽略 `.superpowers/`。
- `git grep ".superpowers"` 只命中 `.gitignore` 和 guardrail skip 列表，没有运行时或规范入口引用。

影响：

- 这是明确的非产品性 LOC 膨胀，约 1363 行。

建议：

- 从版本库删除已跟踪 `.superpowers/sdd/*.md`。
- 若报告需长期保留，迁入 `docs/AI协作/本地Agent/` 或 `docs/review/`。

## P2 / 后续问题

- `audit-reverse-visual` 的 reverse model 证据补全仍在 script，可迁入 `src/writers/html/audit/` 或同级公共 helper。
- G4 扫描 `test/` 时未排除 `test/workspace/`，并发测试时可能遇到临时目录 ENOENT。
- `test/architecture/semantic-model-contract.test.js` 体量大，但当前不建议为了降 LOC 删除；后续可按子规则拆分。

## 对“是否老代码没退役导致大幅增长”的回答

**不是主因。**

证据显示旧公共入口、旧目录、旧 writer-local style compiler、旧 scripts 中的大块算法已经大量退役或迁移；`scripts/` 净删约 547 行，说明不是简单“新旧两套都留着”。

但这轮审计也证明：**当前还不能说完全收口。** 更准确的判断是：

- 旧大模块退役基本完成；
- 新硬化层引入了必要增长；
- 仍存在若干 P1 级半收口问题，会造成继续膨胀或假绿；
- 下一轮应做“收口修复”，不是再大规模新增能力。

## 是否需要沉淀为长期规范

需要，但应先修复再沉淀。建议修复后沉淀四条窄规则：

1. `.superpowers/` 只能是本地 scratch，不得作为版本库长期过程材料。
2. `migration.from` 不能替代 retired lifecycle；称为 retired 的 model path 必须进入 retired registry。
3. audit / E2E / reverse export 的强门禁必须在 `src/` 统一实现，脚本只做参数、路径、输出和退出码。
4. guardrail baseline 不得靠新增 exemption 静默变绿；helper 单一来源不能只靠固定函数名扫描。
