# 任务：行数归因与可删候选审计

## 背景

当前分支相对基线总行数和核心代码均增长。需要一个独立视角做 LOC attribution，列出净增最大的文件并判断是否存在可删、可合并或需要进一步审计的候选。

## 目标

基于 `git diff --numstat` 和文件抽查，给出行数增长的可解释归因，不做代码修改。

## 范围

- 全仓库 diff：`5d4757a6f7efbe8c8410f816d462a90813567fb7..HEAD`
- 重点拆分：`src/`、`scripts/`、`test/`、`docs/`、`.superpowers/`
- 净增最大的前 20 个文件
- 删除最多的前 20 个文件

## 硬约束

- 只读审计，不得修改代码、测试、配置、git index 或 HEAD。
- 只允许写自己的报告：`docs/AI协作/本地Agent/进行中/2026-07-08_架构硬化核心代码膨胀审计/报告_行数归因与可删候选审计.md`
- 不得调用子 agent。
- 不跑真实 InDesign E2E。
- 严格执行项目原则：退役代码必须清理；新增行数只有在换来明确边界、fail-closed、测试覆盖或单一事实源时才合理。

## 建议命令

- `git diff --shortstat 5d4757a6f7efbe8c8410f816d462a90813567fb7..HEAD`
- `git diff --numstat 5d4757a6f7efbe8c8410f816d462a90813567fb7..HEAD`
- 可用脚本汇总各目录净增、最大新增文件、最大删除文件。

## 报告要求

报告必须包含：

- 总体 LOC 结论。
- 按目录归因表。
- 净增最大的前 20 个文件，逐项判定：必要新增 / 可合并 / 可删 / 需人工决定。
- 删除最多的前 20 个文件，说明是否证明旧实现已退役。
- 可删候选总表。
- 是否需要沉淀为长期规范。
