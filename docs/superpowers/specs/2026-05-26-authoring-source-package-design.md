# 作者源码包与反向源码导出设计

## 1. 背景

当前项目已经能把分页 HTML 转成 InDesign，也已经开始把 InDesign 反向导出为 HTML。但现有反向 HTML 更像“视觉还原文件”：它能预览、能对照、能保留文本、图片、PDF、表格和样式，却不适合作为 Agent 长期继续编写的源码。

问题不是 HTML 不能编辑，而是反向 HTML 丢了作者侧的组织方式：

- 页面混在一个大文件里。
- CSS 大量内联到对象上。
- 页面结构、组件、网格变量和 class 意图不够清楚。
- 视觉坐标比作者意图更强，Agent 后续修改时容易越改越乱。

因此需要把“可预览的单个 HTML”与“可继续写作的作者源码包”分开。

本设计补充 `2026-05-25-html-source-structure-preservation-design.md`。前者定义要保留哪些源结构事实；本文定义这些事实如何落成文件结构、反向产物和工具边界。

## 2. 目标

项目应支持两种 HTML 产物：

| 产物 | 主要用途 | 是否适合 Agent 长期编辑 |
| ---- | -------- | ----------------------- |
| 组装后的 `deck.html` | 浏览器预览、HTML -> InDesign 转换入口、视觉对照 | 否，视为生成物 |
| 作者源码包 | Agent 编写、审阅、diff、重构和长期维护 | 是 |

目标是建立统一链路：

```text
作者源码包
-> 组装 deck.html
-> HTML -> InDesign
-> InDesign -> reverse model
-> 反向作者源码包
-> 组装 deck.html
```

这不是“每页一个完整 HTML 网站”。它是一个项目源码包：共享 CSS、共享配置、页面片段和资源清单组合成单个可预览、可转换的 `deck.html`。

## 3. 非目标

本设计不做：

- 不承诺从完全无标签 InDesign 还原出原始作者源码。
- 不还原原始 CSS 的注释、选择器顺序、空行和文件格式。
- 不把页面结构模板变成大量 InDesign 母版。
- 不要求 HTML 作者写反常结构来迁就转换器。
- 不把 `deck.html` 作为人工长期编辑入口。

无标签 InDesign 仍应能导出观察 HTML 和低语义源码包，但不能声称恢复了原作者意图。

## 4. 推荐源码结构

作者源码包目录：

```text
report/
  deck.config.json
  deck.html
  styles/
    tokens.css
    layout.css
    components.css
    pages.css
  pages/
    00-cover.html
    01-agenda.html
    02-masterplan.html
    03-drawing-sheet.html
    04-materials.html
    05-table.html
  assets/
    ...
```

规则：

- `deck.config.json` 是包清单，记录页面顺序、共享样式、资源根、输出模式和目标尺寸。
- `pages/*.html` 是页面片段，每个文件只包含一个或少量 `<section class="page">`，不包含完整 `<html>`、`<head>`、`<body>`。
- `styles/tokens.css` 放颜色、字体、尺寸、层级、常用间距和语义变量。
- `styles/layout.css` 放页面尺寸、页边距、主网格、baseline 和通用布局工具。
- `styles/components.css` 放卡片、指标、图纸框、标注、图例、表格等组件。
- `styles/pages.css` 放少量页面级差异。页面差异过多时，应拆成组件或补页面结构语义。
- `deck.html` 由工具组装生成，供浏览器和转换器使用。人和 Agent 默认不直接编辑它。

## 5. 配置文件

`deck.config.json` 示例：

```json
{
  "schemaVersion": 1,
  "id": "architecture-report",
  "title": "建筑汇报",
  "profile": "architecture-report",
  "unitMode": "presentation",
  "targetSize": "source",
  "entry": "deck.html",
  "styles": [
    "styles/tokens.css",
    "styles/layout.css",
    "styles/components.css",
    "styles/pages.css"
  ],
  "pages": [
    { "id": "cover", "file": "pages/00-cover.html" },
    { "id": "agenda", "file": "pages/01-agenda.html" }
  ],
  "assets": {
    "root": "assets"
  }
}
```

规则：

- `pages` 顺序就是最终页面顺序。
- `id` 必须与页面片段里的 `data-page` 或页面 `id` 可对应。
- `styles` 顺序是级联顺序，不能由组装器随意重排。
- `targetSize` 只描述转换目标，不应改变页面作者源码的视觉意图。

## 6. 页面片段

页面片段示例：

```html
<section class="page"
         id="agenda-page"
         data-page="agenda"
         data-id-semantic="agenda"
         data-id-parent-page="report-parent"
         data-id-layout="contents-grid"
         data-id-margin="14mm 16mm 10mm 18mm"
         data-id-grid="12x6"
         data-id-column-gutter="6mm"
         data-id-row-gutter="8mm"
         data-id-baseline="4mm">
  <h2 class="page-title grid-item"
      data-id-object
      style="--grid-col:1;--grid-span:4;--grid-row:1;--grid-row-span:1">
    汇报结构
  </h2>

  <div class="chapter-card grid-item"
       data-id-object
       data-id-object-style="chapter-card"
       style="--grid-col:5;--grid-span:3;--grid-row:2;--grid-row-span:2">
    <h3>01 轴网与场馆骨架</h3>
    <p>Grid, long-span hall, existing structure.</p>
  </div>
</section>
```

规则：

- 顶层页面必须声明边距和主网格，或能被 CSS 明确推导。
- 顶层可映射对象应尽量使用 `grid-item` 和网格变量。
- 组件内部可以使用局部网格、普通流式排版或内联文字结构。
- PDF、图片、AI、PSD、SVG 等资源仍用自然 HTML 表达，转换层负责置入和预览。
- 不为了方便 InDesign 转换，把正常组件拆成不可读的绝对定位碎片。

## 7. 组装后的 `deck.html`

组装器把源码包生成单个 `deck.html`：

```text
deck.config.json + styles/*.css + pages/*.html -> deck.html
```

组装规则：

- `deck.html` 包含完整 HTML 文档结构。
- CSS 可以内联为多个 `<style data-source-file="...">`，也可以保留 `<link>`；默认建议内联，减少转换时的路径和加载差异。
- 每个页面片段保留来源注释或 `data-id-source-file`，方便反向对照。
- 组装器不得修改页面对象 ID、class、`data-id-*` 或网格变量。
- 组装器可以做路径归一化、资源存在性检查和基础 HTML 校验。

生成文件头应明确：

```html
<!-- Generated by html-indesign. Edit pages/*.html and styles/*.css, not this file. -->
```

## 8. 正向写入 InDesign 的来源信息

HTML -> InDesign 时，除了视觉和语义字段，还应写入源码包来源信息。

文档标签增加：

```json
{
  "kind": "document",
  "sourcePackage": {
    "schemaVersion": 1,
    "config": "deck.config.json",
    "entry": "deck.html",
    "styleFiles": [
      "styles/tokens.css",
      "styles/layout.css",
      "styles/components.css",
      "styles/pages.css"
    ],
    "pageFiles": [
      { "id": "cover", "file": "pages/00-cover.html" },
      { "id": "agenda", "file": "pages/01-agenda.html" }
    ]
  }
}
```

页面标签增加：

```json
{
  "kind": "page",
  "id": "agenda-page",
  "sourceFile": "pages/01-agenda.html",
  "sourceNode": {
    "tagName": "section",
    "id": "agenda-page",
    "classList": ["page"]
  }
}
```

对象标签增加：

```json
{
  "kind": "item",
  "id": "agenda-chapter-01",
  "sourceFile": "pages/01-agenda.html",
  "sourceNode": {
    "tagName": "div",
    "classList": ["chapter-card", "grid-item"],
    "attributes": {
      "data-id-object": "",
      "data-id-object-style": "chapter-card"
    }
  },
  "structure": {
    "parentId": "agenda-page",
    "order": 4,
    "containerPolicy": "group"
  },
  "layout": {
    "grid": { "col": 5, "span": 3, "row": 2, "rowSpan": 2 },
    "cssVars": {
      "--grid-col": "5",
      "--grid-span": "3",
      "--grid-row": "2",
      "--grid-row-span": "2"
    }
  }
}
```

这些字段不是视觉执行补丁，而是反向生成作者源码包的事实源。

## 9. 反向导出产物

InDesign -> HTML 时应同时支持两个产物层级。

### 9.1 视觉产物

用于快速对照和缺陷定位：

```text
reverse-export/
  deck.visual.html
  reverse-model.json
  report.json
  assets/
```

特点：

- 尽量忠实还原 InDesign 视觉。
- 允许绝对定位。
- 保留所有可观察对象。
- 不承诺源码优雅。

### 9.2 作者源码包

用于 Agent 后续继续编写：

```text
reverse-export/
  author/
    deck.config.json
    deck.html
    styles/
      tokens.css
      layout.css
      components.css
      reverse-overrides.css
    pages/
      00-cover.html
      01-agenda.html
      02-masterplan.html
    assets/
      ...
    reports/
      authoring-report.json
      inference-report.json
```

特点：

- `author/deck.html` 仍是生成物。
- `author/pages/*.html` 和 `author/styles/*.css` 是编辑入口。
- `reverse-overrides.css` 只放无法归入 token、布局或组件的残留视觉差异。
- `reports/*.json` 必须说明哪些结构来自确定标签，哪些来自推断，哪些只能视觉保留。

## 10. 反向模式与源码包质量

| 模式 | 是否输出作者源码包 | 源码包质量 | 规则 |
| ---- | ------------------ | ---------- | ---- |
| `authoring` | 必须 | 高 | 依赖完整 `sourcePackage`、`sourceNode`、`structure`、`layout.grid` 标签 |
| `structured` | 应输出 | 中高 | 保留语义和对象，可能局部扁平 |
| `inferred` | 可输出 | 中 | 从母版、参考线、图层、样式、分组和几何推断，必须写置信度 |
| `observation` | 可输出 | 低 | 以视觉保真为主，源码包只作为补标起点 |

`authoring` 模式缺少关键来源标签时必须失败，不得静默降级。

`inferred` 和 `observation` 可以输出源码包，但必须把目录名、报告和 HTML 标记为非确定来源：

```html
<main class="deck observed-deck" data-id-reverse-mode="observation">
```

## 11. 从参考线和对象推断网格

反向作者源码包可以尝试从 InDesign 推断网格，但优先级必须明确：

1. 页面 `html_indesign` 标签中的网格规则。
2. 带 `source="grid"` 或等价协议标签的参考线。
3. 规则性强的参考线族。
4. 页面边距、对象边界和重复模块共同形成的几何证据。

推断步骤：

- 归一化页面尺寸、页边距、参考线和对象边界。
- 对纵向参考线聚类，识别左边距、右边距、列线和栏间距。
- 对横向参考线聚类，识别上边距、下边距、粗行模块和 baseline。
- 检查对象边缘是否贴合网格线或栏间距两侧。
- 计算每个对象的 `grid col/span/row/rowSpan`。
- 低置信度对象保留绝对定位，并写入报告。

规则：

- 对象边缘不能默认反推为主网格。
- 只有当参考线和多个对象共同支持同一套规则时，才输出 `data-id-grid`。
- 推断出的网格必须带 `confidence` 和 `evidence`。

## 12. Agent 编写建议

Agent 最佳编写方式：

- 编辑 `pages/*.html` 和 `styles/*.css`。
- 用共享 token 和组件 class 复用样式。
- 每页声明主网格，顶层对象贴合主网格。
- 复杂页面先定义区域，再在区域内做卡片阵列、图纸区、指标区或表格区。
- 用 `deck.html` 做预览和转换入口，但不把它当源码。

这种方式不会限制 Agent 做 4x4、6x4 或更复杂的卡片阵列。限制只来自页面区域、主网格和组件语义，目的是让排版可读、可验、可回读。

## 13. 工具边界

应新增或扩展三类工具。

| 工具 | 输入 | 输出 | 责任 |
| ---- | ---- | ---- | ---- |
| 作者源码组装器 | `deck.config.json` | `deck.html` | 合并样式和页面片段，校验路径和页面顺序 |
| 作者规则检查器 | 源码包或 `deck.html` | 报告 | 检查页面网格、语义 token、资源路径、生成物是否被手改 |
| 反向作者写出器 | `reverse-model.json` | `author/` 源码包 | 按标签、结构和网格生成多文件源码 |

现有 `npm run lint:authoring -- <deck.html>` 可以继续支持单文件输入，但长期应支持：

```powershell
npm run lint:authoring -- --package report/deck.config.json
```

## 14. 校验规则

新增检查项：

| 检查 | 等级 | 说明 |
| ---- | ---- | ---- |
| `AUTHOR_PACKAGE_CONFIG_MISSING` | error | 源码包缺少 `deck.config.json` |
| `AUTHOR_PAGE_FILE_MISSING` | error | 配置引用的页面片段不存在 |
| `AUTHOR_STYLE_FILE_MISSING` | error | 配置引用的样式文件不存在 |
| `AUTHOR_GENERATED_ENTRY_DIRTY` | warning/error | `deck.html` 与源码包重新组装结果不一致 |
| `AUTHOR_PAGE_FRAGMENT_INVALID` | error | 页面片段不是合法页面容器 |
| `SOURCE_FILE_LABEL_MISSING` | warning/error | InDesign 标签缺少来源文件 |
| `REVERSE_AUTHOR_GRID_INFER_LOW_CONFIDENCE` | warning | 反向网格推断置信度低 |
| `REVERSE_AUTHOR_ABSOLUTE_FALLBACK` | warning | 对象无法落到网格，只能绝对定位 |

`strict` 模式下，生成物不一致、核心对象缺标签、低置信度网格推断应阻止交付。

## 15. 实施顺序建议

后续 plan 应按以下顺序拆：

1. 定义 `deck.config.json` schema 和源码包组装器。
2. 把现有架构汇报 fixture 拆成源码包，并由组装器生成当前 `deck.html`。
3. 扩展前向快照和标签写入，记录 `sourcePackage`、`sourceFile`、`sourceNode`、`structure` 和 `layout.grid`。
4. 新增反向作者写出器，先支持完整标签的 `authoring` 模式。
5. 给 `structured`、`inferred`、`observation` 模式输出低等级源码包和报告。
6. 把规则检查器扩展到源码包入口。

每一步都必须保留当前单文件 `deck.html` 转换入口，避免破坏现有 E2E。

## 16. 成功标准

完成后应满足：

- Agent 可以在多文件源码包中自然编写建筑汇报页面。
- 组装出的 `deck.html` 与现有浏览器预览和 HTML -> InDesign 转换兼容。
- 正向生成的 InDesign 带有足够来源标签，可反向导出 `author/` 源码包。
- 反向源码包恢复页面顺序、共享样式、页面片段、主网格、class、`data-id-*` 和资源引用。
- 无标签 InDesign 仍能导出视觉 HTML，并可生成低语义源码包作为补标起点。
- 视觉对照产物和作者源码包职责分离，不再把一个巨大反向 HTML 同时当预览、源码和迁移报告。

## 17. 关键决策

- 作者源码包是长期编辑入口，`deck.html` 是生成物。
- 转换器仍消费组装后的 `deck.html`，不要求 InDesign 执行层理解多文件源码。
- 反向导出必须同时服务视觉对照和后续编辑，二者用不同产物表达。
- 完整作者源码恢复依赖正向写入的来源标签。
- 无标签 InDesign 不能恢复原作者源码，但可以生成可补标的低语义源码包。
- 网格推断可以做，但必须有置信度；对象边缘不能伪装成主网格。
