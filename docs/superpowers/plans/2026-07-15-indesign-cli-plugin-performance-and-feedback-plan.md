# InDesign CLI 与 HTML 插件可靠构建和保真反馈轻量实施计划

**目标：** 让 Agent 写出的合规 HTML 在正式构建时得到可信结论：失败立即停止并说明原因；成功不仅代表文件生成，还代表原 HTML 与真实 InDesign 对象在项目支持范围内完成核对。

## 最终流程

正式构建默认执行：

```text
严格检查
→ 编译
→ 构建真实 InDesign 文档
→ 读取刚构建文档的对象快照
→ 与原 HTML 的统一语义事实逐项核对
→ 核对通过后导出 INDD、PDF 和 IDML
→ 返回“已验证”结果
```

任何一步失败都立即停止，关闭且只关闭本次构建创建的临时文档，并把具体问题返回给 Agent。系统不继续执行后续步骤，也不使用相同参数自动重试；Agent 修改 HTML 后再重新发起。

保留显式的快速草稿模式。草稿可以跳过构建后的保真核对，但结果必须明确标记为“尚未验证”，不能作为正式交付成功。

## 实施边界

- 仍只保留四个正式工具：`html.authoring_lint`、`html.compile_instructions`、`html.build_indesign`、`html.reverse_export`。
- 正式保真核对由 `html.build_indesign` 内部完成，不把 E2E、结构 diff、视觉 diff 或诊断脚本增加到 Agent 工具列表。
- HTML 和 InDesign 都先转换为统一语义事实，再比较协议注册表和格式能力矩阵声明为可双向保存的字段；不直接比较 HTML 字符串和 INDD 文件。
- 内容、语义和可信作者结构不允许丢失；位置、尺寸、裁切、缩放及其他数值字段按项目已有容差判断。
- 不支持的 HTML 能力必须在严格检查或编译阶段明确失败，不能等到构建后静默丢弃。
- 原位资源仍按 UNC 引用规则处理，不为核对过程复制原始事务所素材。
- 本轮不实现原 HTML 与导出 PDF 的图片视觉对比。
- 本轮不运行普通交付的二次回环；二次回环继续只作为库级稳定性验证。
- 本轮不实现常驻插件进程、十秒心跳、细分到每个 InDesign 内部动作的性能平台，也不预设未经测量的执行器优化。

## 1. 构建失败立即停止

`mcp-indesign` 按依赖顺序一次只执行一个动作。构建成功后才允许读取实际文档，保真核对成功后才允许正式导出，导出成功后才验证产物。

CLI 必须把脚本异常、超时、返回失败、缺少产物和嵌套结果中的失败统一视为硬错误。启动器直接呈现底层错误，不再只给 Agent 一个笼统的“子命令失败”。

失败结果至少说明：

- 失败发生在哪个阶段；
- 错误代码和原因；
- 涉及的页面、对象或文件；
- 已生成的诊断产物；
- Agent 下一步应该修改什么；
- 当前错误不可原样自动重试。

构建脚本必须记录本次新建的文档。失败时只关闭该文档，不影响用户原本打开的 InDesign 文件，也不留下空白或半成品文档。

## 2. 正式保真核对

HTML 编译阶段保留一份“预期事实”，包括页面、对象、文字、资源、语义、结构、网格、位置、尺寸、样式和置入参数。真实 InDesign 构建完成后，在当前打开的文档上直接读取对象快照，转换成同一套统一语义事实后再比较，避免关闭后重新打开。

正式构建必须核对：

- 页面数量、顺序和尺寸；
- 对象身份、层级、标签和可信作者结构；
- 文字、字符样式、表格和母版家具；
- 图片、PDF、图纸等资源引用；
- 对象位置、大小、网格关系、裁切、缩放和适配方式；
- 协议声明可双向保存的字体、填充、描边、圆角等样式；
- 缺字体、文字溢出、资源断开和不支持能力。

已有的源码回环、内容库存、结构签名、可信结构保护和有效差异能力继续作为共享判断依据，不另造一套门禁。当前首轮内容库存没有严格核对几何字段，正式模式需要补齐这部分，并按协议能力过滤不能合法比较的字段。

通过时返回“已验证”、核对摘要和正式产物。失败时返回差异报告，至少指出页码、对象、字段、预期值和实际值；不能因为 INDD 或 PDF 文件已经存在就报告成功。

## 3. Agent 可见反馈与启动开销

长任务只需要显示“正在检查、正在构建、正在核对、正在导出”及完成或失败，不建设额外心跳系统。最终结果统一带回错误、警告、产物和各主要阶段耗时。

已安装版本与远端版本相同时，普通命令只检查当前状态和关键入口文件；完整 CLI、Node、winax、插件和 Edge 检查仅在首次安装、运行环境升级或显式深度健康检查时执行。自动更新清单仍在普通命令前检查。

插件进程常驻和更细的 InDesign 性能优化暂缓，只有后续测量证明它们是主要耗时来源时再单独设计。

## 4. Skill 与发布

`html.authoring_lint` 保留为创作过程中的快速自检，方便 Agent 在打开 InDesign 前尽早修改 HTML。正式构建不能假设 Agent 已经单独检查过，必须在内部执行同一套严格检查，再继续编译和保真核对。

统一 Skill 的正常交付流程更新为“组装作者包 → 一次正式构建”。Agent 可以在创作过程中按需使用快速自检，但正式交付不需要手工拼接内部 E2E 或多个审核脚本。

`html.compile_instructions` 只用于不打开 InDesign 的排查。`html.reverse_export` 继续用于读取既有 InDesign、重建人类文档或用户明确要求的独立反向导出，不作为普通 HTML 制作流程中的手工补充步骤。

Skill 必须告诉 Agent：构建失败后先读取具体差异并修改 HTML，不得用相同参数连续重试，也不得自行追加二次回环。

实施完成后，`indesign-cli` 升级为 `0.5.2`，`@sa/html-indesign` 升级为 `0.2.2`。统一运行环境继续按既有 NAS/GitHub 流程发布，统一 Skill 继续通过公司画廊单独发布。

## 完成标准

- 正式构建失败后不会继续核对、导出或验证，也不会留下本次创建的空白文档。
- 正式构建只有在真实 InDesign 对象与原 HTML 的受支持事实核对通过后才返回“已验证”。
- 草稿结果始终明确标记为未验证。
- Agent 能直接看到具体页面、对象、字段和修改方向。
- 普通命令在没有更新时不再重复完整运行环境深检。
- 普通交付不运行二次回环，也不执行 HTML 与 PDF 的图片视觉对比。
